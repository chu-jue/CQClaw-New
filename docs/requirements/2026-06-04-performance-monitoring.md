# CQClaw 手机性能监控需求

## 状态

- 文档日期：2026-06-04
- 当前状态：需求已记录，暂缓实施
- 后续起点：补充 API 与测试设计，编写实施计划后再进入开发

## 背景与目标

CQClaw 需要新增独立的“性能监控”导航页，用于持续观察 Android
设备整机、当前前台 App 和一个用户指定 App 的 CPU 与内存使用情况。

监控由 CQClaw 服务端在后台持续执行。用户切换页面或关闭性能监控页后，
活动会话仍继续采样；用户手动停止时生成完整历史记录。

## 已确认范围

### 监控对象

每个监控会话绑定一台 Android 设备，同时记录：

1. 整机 CPU 和内存使用情况。
2. 自动识别并持续跟踪当前前台 App。
3. 一个由用户选择的固定目标 App。

系统允许不同设备同时运行独立会话，但同一设备同时只允许一个活动会话。

### 会话控制

- 用户手动开始和停止监控。
- 离开性能监控页面后，服务端继续后台采样。
- 采样间隔支持 `1 / 2 / 5 / 10` 秒，默认 `2` 秒。
- 手机断开时，会话进入“等待重连”；设备重连后自动继续。
- 指定 App 未运行时记录空值，继续采集整机和前台 App 数据。
- 服务停止或重启时保留已采样数据，并将未正常停止的会话标记为“意外中断”。

### 性能指标

每个采样点应尽量包含：

- 整机 CPU 使用率。
- 整机总内存、已用内存、可用内存和使用率。
- 当前前台 App 包名及切换事件。
- 前台 App CPU 使用率和内存占用。
- 指定 App CPU 使用率和内存占用。
- App 内存数据来源，例如 PSS 或降级后的 RSS。
- 单项采样错误和降级状态。

整机 CPU 优先通过 `/proc/stat` 两次采样差值计算。整机内存通过
`/proc/meminfo` 计算。App CPU 应汇总目标包名相关进程，App 内存优先使用
PSS，失败时允许降级为 RSS。需要兼容不同 Android 版本和 ROM 的输出差异。

单项采样失败不能终止整个监控会话。

## 页面与交互

### 导航

在“设备”后新增“性能监控”导航 Tab，页面路径：

```text
/performance-monitor.html
```

导航使用项目已有的 `activity` 图标。

### 页面区域

性能监控页包含以下区域：

1. **会话控制栏**
   - 选择在线设备。
   - 搜索并选择一个指定 App。
   - 展示当前前台 App。
   - 选择采样间隔。
   - 配置告警阈值。
   - 开始或停止监控。
   - 显示后台运行状态和持续时间。

2. **实时指标区**
   - 整机、前台 App、指定 App 的当前值、平均值和峰值。
   - 展示数据来源、连接状态和告警状态。

3. **实时趋势区**
   - CPU 趋势图：整机、前台 App、指定 App。
   - 内存趋势图：整机使用率、前台 App PSS/RSS、指定 App PSS/RSS。
   - 标记告警点、告警阈值和前台 App 切换时间点。
   - 支持实时、最近 1 分钟、最近 5 分钟和全部数据范围。

4. **运行会话与历史区**
   - 展示所有后台运行会话。
   - 可进入其他设备的活动会话或停止会话。
   - 展示最近历史记录并直接预览。

页面必须使用现有主题变量，并验证浅色与深色主题。

## 图表方案

采用本地内置的 Apache ECharts，不使用 CDN，确保离线环境可用。

建议资源路径：

```text
static/vendor/echarts/echarts.min.js
```

图表应支持：

- 实时动态更新。
- Tooltip 采样详情。
- 图例显示与隐藏。
- `dataZoom` 时间范围缩放。
- `markPoint` 告警和峰值标记。
- `markLine` 阈值标记。
- 前台 App 切换事件标记。
- 浅色和深色主题动态配色。

## 告警

用户可以在开始监控前配置告警阈值。建议默认值：

| 指标 | 默认阈值 |
| --- | ---: |
| 整机 CPU 使用率 | 85% |
| 整机内存使用率 | 85% |
| App CPU 使用率 | 50% |
| App PSS/RSS | 500 MB |
| 连续超限次数 | 3 次 |

前台 App 和指定 App 均应用 App 告警规则。

连续超过阈值达到配置次数后触发一次告警；恢复到阈值以下时记录恢复事件。
不能为每个超限采样点重复生成相同告警。

## 历史记录与资源中心

每个会话使用独立 JSONL 文件增量写入，建议路径：

```text
data/performance-sessions/<session-id>.jsonl
```

记录内容包括：

- 会话元数据：设备、指定 App、采样间隔、告警配置和开始时间。
- 每个采样点。
- 前台 App 切换事件。
- 告警触发和恢复事件。
- 停止后的摘要：持续时间、平均值、峰值和告警次数。

增量写入用于降低服务异常退出造成的历史数据丢失。

资源中心新增“性能监控记录”分类，支持：

- 查看文件数量、占用空间和最近修改时间。
- 打开性能历史目录。
- 预览历史会话摘要和趋势图。
- 删除单个历史会话。
- 按时间批量清理历史会话。
- 保护正在运行的会话，禁止预览清理或删除流程误删。

## 推荐架构

采用“服务端后台采样会话”架构：

- Python 服务端管理活动会话和采样线程。
- 每个活动会话独立维护设备、目标 App、阈值、统计摘要和历史文件。
- 页面通过 API 创建、查询、停止和轮询会话。
- 页面刷新后可重新连接到已有后台会话。
- 资源中心复用性能历史解析和 ECharts 预览组件。

不采用浏览器页面直接采样，因为页面关闭或切换会导致监控中断。

实施前还需要完成以下设计：

- 性能会话 API 路径、请求参数和响应结构。

- 自动化测试、静态页面测试与真实设备验证方案。

## 验收标准

- 导航中可进入独立性能监控页。
- 可选择设备、指定 App、采样间隔和告警阈值并开始监控。
- 页面离开后会话继续运行，返回后可恢复查看。
- 实时展示整机、前台 App 和指定 App 的 CPU/内存数据。
- 图表能清晰展示趋势、告警和前台 App 切换事件。
- 支持多设备独立活动会话，同一设备不能重复启动会话。
- 停止或异常中断后均保留可读取的历史记录。
- 资源中心可以预览和删除性能历史，且不能误删活动会话。
- 浅色、深色主题及常见桌面窗口尺寸均可正常使用。
- 单项采样失败或设备短暂断连不会导致服务或其他会话崩溃。
